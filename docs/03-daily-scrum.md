## デイリースクラムの目的

### 主な目的
1. **進捗の確認**: スプリントゴールに向けた進捗を確認
2. **計画の調整**: 必要に応じて作業計画を調整
3. **障害の特定**: 進捗を妨げる障害を早期発見
4. **チームの連携**: チームメンバー間の情報共有

### デイリースクラムの特徴
- **時間**: 最大15分
- **頻度**: 毎日同じ時間
- **参加者**: 開発チーム（必須）、スクラムマスター、プロダクトオーナー（任意）
- **形式**: 立って行う（オンラインの場合は画面共有）

## 効果的なデイリースクラムの進め方

### 3つの質問

#### 1. 昨日何を達成したか？
**目的**: 進捗の確認と成果の共有

**良い回答例**:
- 「ユーザー登録フォームのバリデーション機能を実装完了した」
- 「検索APIの単体テストを完了した」
- 「デザインのフィードバックを反映してUIを更新した」

**避けるべき回答**:
- 「コードを書いた」
- 「会議に参加した」
- 「調査をした」

#### 2. 今日何をするか？
**目的**: 今日の計画とチーム内での調整

**良い回答例**:
- 「検索結果の表示機能を実装する」
- 「昨日実装した機能の統合テストを行う」
- 「プロダクトオーナーと受け入れ基準を確認する」

**避けるべき回答**:
- 「できるだけ進める」
- 「何かやる」
- 「バグを直す」

#### 3. 何か障害があるか？
**目的**: 障害の早期発見と解決

**報告すべき障害**:
- 技術的な問題
- 外部依存関係の遅延
- 情報不足
- 環境の問題

**障害の報告方法**:
- 具体的な内容
- 影響度
- 必要な支援

### デイリースクラムの進行

#### 開始前の準備
1. **時間の設定**: 毎日同じ時間（例：9:00）
2. **場所の確保**: チームが集まれる場所
3. **ツールの準備**: オンラインの場合は画面共有ツール
4. **ボードの準備**: カンバンボードまたはデジタルツール

#### 進行の流れ
1. **開始**: 時間通りに開始
2. **順番の決定**: 毎回同じ順番またはランダム
3. **各メンバーの報告**: 3つの質問に回答
4. **質問と議論**: 必要に応じて簡潔に
5. **終了**: 15分以内で終了

#### ファシリテーションのポイント
- **時間管理**: 15分を厳守
- **全員参加**: 全員が発言する機会を確保
- **焦点維持**: スプリントゴールに集中
- **障害の記録**: 障害は別途対応

## 進捗の可視化

### カンバンボードの活用

#### 基本的な列構成
```
To Do → In Progress → Done
```

#### 詳細な列構成
```
Backlog → To Do → In Progress → Review → Done
```

#### カードの情報
- **タイトル**: タスク名
- **担当者**: 責任者
- **見積もり**: ストーリーポイント
- **残り時間**: 残り作業時間
- **優先度**: 高/中/低
- **タグ**: カテゴリ分け

### バーンダウンチャート

#### 作成方法
1. **縦軸**: 残り作業量（ストーリーポイント）
2. **横軸**: スプリント日数
3. **理想線**: 直線的な減少
4. **実際線**: 実際の進捗

#### 分析ポイント
- **理想線との乖離**: 計画の精度
- **急激な変化**: 障害の発生
- **終盤の停滞**: 見積もりの問題

### ベロシティの測定

#### ベロシティの計算
```
ベロシティ = 完了したストーリーポイントの合計
```

#### ベロシティの活用
- **計画の精度向上**: 次回スプリントの見積もり
- **チームパフォーマンス**: 改善の指標
- **キャパシティ管理**: 適切な作業量の設定

## 障害の早期発見と解決

### 障害の分類

#### 技術的障害
- **環境の問題**: 開発環境の不具合
- **技術的負債**: 既存コードの問題
- **外部依存**: APIやライブラリの問題

#### プロセス的障害
- **情報不足**: 要件の不明確さ
- **承認待ち**: 意思決定の遅延
- **リソース不足**: 人材やツールの不足

#### 人的障害
- **スキル不足**: 必要な技術の不足
- **コミュニケーション**: チーム内の意思疎通
- **モチベーション**: チームメンバーの意欲

### 障害管理のプロセス

#### 1. 障害の記録
**記録項目**:
- 障害の内容
- 発見日時
- 影響度（高/中/低）
- 担当者
- 解決期限

#### 2. 優先順位付け
**優先度の基準**:
- スプリントゴールへの影響
- 他のタスクへの影響
- 解決の緊急性

#### 3. 解決策の検討
**解決方法**:
- チーム内での解決
- スクラムマスターの支援
- 外部リソースの活用

#### 4. フォローアップ
**確認事項**:
- 解決状況の確認
- 再発防止策の検討
- プロセスの改善

### 障害解決のベストプラクティス

#### 1. 早期発見
- デイリースクラムでの定期的な確認
- チームメンバーの積極的な報告
- 進捗の可視化による異常の早期発見

#### 2. 迅速な対応
- 優先度に応じた迅速な対応
- 適切なリソースの投入
- 定期的な状況確認

#### 3. 根本原因の特定
- 表面的な解決ではなく根本原因の特定
- 再発防止策の検討
- プロセスの改善

## ベストプラクティス

### 1. 時間管理
- **15分厳守**: 時間を超過しない
- **準備**: 事前に報告内容を整理
- **効率化**: 不要な議論は別途実施

### 2. 全員参加
- **順番の決定**: 全員が発言する機会を確保
- **質問の奨励**: 不明点は積極的に質問
- **支援の申し出**: 困っているメンバーへの支援

### 3. 焦点の維持
- **スプリントゴール**: 常にスプリントゴールを意識
- **具体的な報告**: 抽象的な報告を避ける
- **進捗の確認**: 実際の進捗を確認

### 4. 継続的改善
- **振り返り**: デイリースクラムの効果を定期的に評価
- **改善**: 問題点を特定し改善
- **学習**: 良い実践を共有

### よくある問題と解決策

#### 問題1: 時間が超過する
**原因**:
- 詳細な議論が発生
- 技術的な問題の解決を試みる
- 全員が発言しない

**解決策**:
- 議論は別途実施
- 技術的問題は別会議で対応
- 時間管理を徹底

#### 問題2: 進捗が不明確
**原因**:
- 抽象的な報告
- 具体的な成果物が不明
- 見積もりが不正確

**解決策**:
- 具体的な成果物を報告
- 完了基準を明確にする
- 見積もり精度を改善

#### 問題3: 障害が解決されない
**原因**:
- 障害の記録が不十分
- 責任者が不明確
- フォローアップが不十分

**解決策**:
- 障害の詳細を記録
- 責任者を明確にする
- 定期的なフォローアップ

## テンプレート

### デイリースクラムアジェンダ
```
1. 開始 (0分)
2. 各メンバーの報告 (10分)
   - 昨日の成果
   - 今日の計画
   - 障害の報告
3. 質問と議論 (3分)
4. 終了 (2分)
```

### 障害記録テンプレート
| ID | 障害内容 | 発見日 | 影響度 | 担当者 | 解決期限 | ステータス |
|----|----------|--------|--------|--------|----------|-----------|
| 1 | 開発環境の不具合 | 2024/1/1 | 高 | 田中 | 2024/1/2 | 解決済み |
| 2 | API仕様の不明確 | 2024/1/3 | 中 | 佐藤 | 2024/1/5 | 対応中 |

### 進捗報告テンプレート
```
【昨日の成果】
- 完了したタスク: [具体的なタスク名]
- 進捗: [完了率または残り時間]

【今日の計画】
- 実施予定タスク: [具体的なタスク名]
- 目標: [今日の目標]

【障害】
- 障害内容: [具体的な内容]
- 影響度: [高/中/低]
- 必要な支援: [具体的な支援内容]
```
